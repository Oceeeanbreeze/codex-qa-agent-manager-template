# Пошаговая настройка с помощью Codex

## Цель
Использовать Codex как guided operator assistant, чтобы систему можно было воспроизвести шаг за шагом, даже если полная установка не укладывается в одну команду или один prompt.

## Зачем нужен этот режим
Реальная настройка почти всегда состоит из двух частей:
- детерминированный repository bootstrap, который можно документировать и частично скриптовать;
- локальная донастройка runtime, зависящая от машины, установленных инструментов, shell behavior, permissions и безопасных локальных значений.

Этот документ задает staged setup model, в котором оператор и Codex двигаются фазами, пока установка не совпадет с нужной system shape и runtime behavior.

## Базовое правило
Настройка должна идти по фазам.
Codex не должен пытаться "сделать все сразу".
Каждая фаза заканчивается одним из состояний:
- green и можно идти дальше;
- blocked с конкретным следующим фикс-шагом;
- ожидание локального значения, которое может безопасно дать только оператор.

## Кто за что отвечает

### Ответственность оператора
- клонировать репозиторий;
- запускать локальные команды, когда требуется выполнение;
- устанавливать локальные prerequisites, например Python или Ollama;
- задавать безопасные локальные значения для runtime paths и credentials;
- подтверждать sensitive actions;
- отмечать завершение фазы.

### Ответственность Codex
- читать опубликованные system files и сохранять intended role model;
- инспектировать текущее состояние workspace;
- объяснять, что уже настроено и чего не хватает;
- вести оператора по фазам;
- обновлять безопасные локальные файлы внутри workspace по запросу;
- не скрывать блокеры;
- проверять, что итоговая установка соответствует задокументированной системе.

## Фазовая модель

### Фаза 0. Понимание репозитория
Цель:
- проверить, что Codex может прочитать опубликованные agent-system files и пересказать архитектуру.

Готово, когда:
- Codex правильно определяет role model, routing model, memory model и operator docs.

### Фаза 1. Bootstrap и alignment manifest
Цель:
- создать local workspace skeleton;
- создать `memory/config.yaml`;
- создать `configs/runtime-manifest.local.yaml`;
- выровнять manifest со значениями локальной машины.

Готово, когда:
- bootstrap завершен;
- оператор достаточно заполнил manifest для runtime checks.

### Фаза 2. Runtime prerequisites
Цель:
- проверить, что обязательные локальные инструменты установлены и реально доступны из operator shell.

Типовые проверки:
- Git;
- PowerShell;
- Python launcher;
- Ollama или другой embedding provider;
- наличие embedding model;
- writable vault и storage paths.

Основные ссылки:
- `docs/ru/RUNTIME_INSTALLATION.md`
- `tools/install-runtime-prereqs.ps1`

Готово, когда:
- `doctor` зеленый или оставляет только понятные warnings.

### Фаза 3. Готовность memory loop
Цель:
- подтвердить, что система не просто хранит файлы;
- она должна уметь читать, индексировать, искать и архивировать.

Типовые проверки:
- `health`;
- `preflight`;
- `search`;
- `index`;
- `finalize`.

Готово, когда:
- memory loop работает или единственные недостающие части явно описаны как локальные scripts.

### Фаза 4. Готовность оркестрации Codex
Цель:
- подтвердить, что Codex инициализируется от самого репозитория, а не от скрытого прошлого контекста.

Готово, когда:
- bootstrap prompt работает;
- Codex описывает те же routed roles и operating rules, что и docs.

### Фаза 5. Battle-ready review
Цель:
- совместно подтвердить runtime, operator, security и parity readiness.

Готово, когда:
- `docs/BATTLE_READY_CHECKLIST.md` зеленый.

### Фаза 6. Доказательство публичной воспроизводимости
Цель:
- доказать, что тот же репозиторий приводит к тому же результату на fresh clean clone.

Готово, когда:
- `docs/GIT_RELEASE_AND_PARITY_CHECKLIST.md` зеленый.

## Рекомендуемый паттерн работы в чате

### Паттерн A. Фазой управляет Codex
Используй, когда хочешь, чтобы Codex сам оценил workspace и сказал следующий точный шаг.

Codex должен:
- проверить статус текущей фазы;
- делать только минимальные безопасные изменения;
- давать точную следующую команду, если требуется локальный запуск;
- останавливаться после summary по фазе.

### Паттерн B. Фазой управляет оператор
Используй, когда ты уже знаешь нужную команду и хочешь, чтобы Codex потом интерпретировал результат.

Codex должен:
- объяснить output команды;
- сказать, pass это, warn или fail;
- предложить следующий узкий фикс.

## Как избежать setup drift
- держи `configs/runtime-manifest.local.yaml` как локальный source of truth для runtime choices;
- держи `memory/config.yaml` синхронизированным с manifest;
- не позволяй Codex придумывать hidden defaults, которые не отражены в файлах;
- после каждой значимой фазы обновляй manifest или релевантный config file.

## Когда Codex должен остановиться и спросить
Codex должен поставить паузу, когда:
- нужен secret, token или credential;
- path нельзя безопасно вывести автоматически;
- инструмент нужно установить вне repo;
- требуется destructive или approval-gated action;
- может быть затронут production или другое sensitive environment.

## Что обязательно должно совпасть для формулировки "развернуто как у меня"
К концу staged setup установка должна совпадать по:
- той же role chain и routing rules;
- той же memory shape и role-scoped boundaries;
- тому же operator command vocabulary;
- тем же health и doctor gates;
- той же production-disabled-by-default posture;
- той же battle-ready и release parity discipline.

## Использование вместе с prompt pack
Используй `docs/ru/SETUP_CHAT_PROMPTS.md`, чтобы проходить фазы готовыми Codex prompts.
